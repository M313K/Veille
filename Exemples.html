Exemple d'Application Web sur Azure App Service

Voici une vue d'ensemble du processus et des composants qui interviennent dans un scénario d'authentification et de traitement de requêtes dans Azure App Service :

Workflow

Un utilisateur émet une requête HTTPS vers le domaine par défaut de l'App Service sur azurewebsites.net. Ce domaine pointe automatiquement vers l'IP publique intégrée de votre App Service.

  • La connexion TLS est établie depuis le client directement vers l'App Service. La certification est entièrement gérée par Azure.
  • Easy Auth, une fonctionnalité d'Azure App Service, garantit que l'utilisateur qui accède au site est authentifié avec Microsoft Entra ID.
  • Le code de votre application déployé sur App Service traite la requête, par exemple en se connectant à une instance d'Azure SQL Database, à l'aide d'une chaîne de connexion configurée comme paramètre d'application.
  • Les informations relatives à la requête originale adressée à App Service et à l'appel à Azure SQL Database sont journalisées dans Application Insights.

Composants

Voici un aperçu des composants clés dans cette architecture :

  • Microsoft Entra ID : Un service de gestion des identités et des accès basé sur le cloud, fournissant un plan de contrôle d’identité pour gérer les autorisations et les rôles des utilisateurs qui accèdent à votre application web.
  • App Service : Une plateforme managée permettant de créer, déployer et mettre à l’échelle des applications web avec une gestion simplifiée et un haut niveau d’automatisation.
  • Azure Monitor : Un service de supervision qui collecte, analyse et agit sur les données de télémétrie dans votre déploiement, permettant de surveiller la santé et les performances de votre application.
  • Azure SQL Database : Un service de base de données relationnelle managée, permettant de stocker et de gérer des données relationnelles avec une haute disponibilité et une sécurité renforcée.

Extension vers la Haute Disponibilité et la Résilience

Pour étendre l'architecture initiale vers un environnement à haute disponibilité, nous pouvons déployer l'application dans plusieurs régions. Cela permet de garantir que, même en cas de panne d'une région, le trafic peut être redirigé automatiquement vers une autre région, assurant ainsi la continuité des services.

Dans cette architecture étendue, les composants sont répartis sur plusieurs régions, et Azure Front Door permet de gérer le routage du trafic en fonction de la région disponible, tout en optimisant les performances et en offrant une protection grâce au pare-feu d'application Web (WAF).

Haute Disponibilité

Pour assurer une disponibilité continue de votre application, elle est déployée dans deux régions distinctes : une région primaire et une région secondaire. En cas de panne de la région primaire, Azure redirige automatiquement le trafic vers la région secondaire, garantissant ainsi la disponibilité constante de votre application.

Azure Front Door

Azure Front Door joue un rôle clé dans l'optimisation du routage du trafic entre les régions. Grâce à son mécanisme de mise en cache, il réduit la latence et améliore les performances. En plus de cela, il protège votre application avec un pare-feu d'application Web (WAF), offrant ainsi une sécurité renforcée contre les menaces externes.

Géoréplication des Données

La géoréplication est essentielle pour garantir la résilience des données de votre application. Les services tels que Azure SQL Database et Azure Cosmos DB sont répliqués entre plusieurs régions. Cela permet non seulement d'assurer la continuité des données en cas de défaillance d'une région, mais aussi de garantir une performance optimale en fonction de la localisation des utilisateurs.
